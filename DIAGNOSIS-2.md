## 5. Why-aren't-we-on-the-right-build hypotheses (ranked)

### H1 — Vercel hasn't built the fix-branch preview yet, OR the build failed ⭐ MOST LIKELY

You opened the preview within minutes of the push. Vercel's queue and build can take 1–3 minutes (sometimes longer if other deploys are queued). If the build fails, Vercel may still serve a fallback that doesn't reflect your branch.

**Verify:** Vercel dashboard → `nyayiq-app` project → Deployments tab. Find the deploy for branch `fix/login-cookie-bug`. Check:
- Status: **Building / Ready / Errored / Canceled**?
- If Errored, the build log is the next thing to read.
- The commit hash shown should be `331a71d`.

### H2 — You tested production (`nyayiq.in`) instead of the preview URL ⭐ LIKELY

Production is on `main`, which has `f099126`. If `nyayiq.in` is somehow still serving `720a141`, that means main has not been redeployed since the merge of `f099126` two weeks ago — possible if the project's "Production Branch" or auto-deploy is misconfigured.

**Verify:** What URL are you actually testing? Paste it back to me. The preview URL pattern is `https://nyayiq-app-git-fix-login-cookie-bug-<vercel-account>.vercel.app/`. The production URL is `https://nyayiq.in/`.

### H3 — Old preview URL from a prior deployment (LOW)

Vercel keeps a per-deployment URL like `https://nyayiq-app-<sha>-<account>.vercel.app/`. If you bookmarked or revisited a previous one of those, it's frozen on its specific commit forever. The branch alias `nyayiq-app-git-fix-login-cookie-bug-...` should always point to the latest build of that branch — but a per-deployment URL won't.

**Verify:** Does the URL contain `git-fix-login-cookie-bug` (branch alias, latest), or a SHA-like fragment (frozen)?

### H4 — Vercel project is configured to NOT auto-deploy feature branches (LOW)

Default Vercel behavior is to preview every branch, but project settings can restrict to `main` only. If preview deploys are off, no `fix/login-cookie-bug` build exists at all.

**Verify:** Vercel dashboard → Settings → Git → "Preview Deployments". Should be enabled.

### H5 — Browser-cached HTML from an earlier visit (LOW)

If you previously visited `/app` and got the broken page, the browser may be serving cached HTML.

**Verify:** Hard refresh (Ctrl-Shift-R / Cmd-Shift-R), or DevTools → Network → "Disable cache" while reloading.

### H6 — The bug really IS live on `331a71d` despite source being clean (VERY UNLIKELY)

This would require one of:
- Our local branch and origin/`fix/login-cookie-bug` differ (verified they don't — both at `331a71d`)
- Next 16 has a build cache that re-emits old `app/(app)/app/page.tsx` (no published behavior like this; would be a Next.js bug)
- A third-party plugin/transform injects this input at build time (no such tool visible in `package.json`)

Only worth investigating if H1–H5 all rule out and we have proof Vercel is building `331a71d` and serving the broken page.
